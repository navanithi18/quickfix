A2-Config Files 

Frappe uses configuration files like common_site_config.json and site_config.json to store system and site-level settings. common_site_config.json contains configurations shared across all sites in a bench, such as Redis connections and background worker settings. site_config.json stores configuration specific to a single site, like database credentials and site-specific keys. If a secret (API keys, passwords, tokens) is accidentally placed in common_site_config.json, it becomes accessible to all sites in the bench and may get committed to version control, creating a serious security risk.

A2-Bench Start Processes

When running bench start, four main processes are launched: web, worker, scheduler, and socketio. The web process handles HTTP requests from users, while the worker executes background jobs queued by the system. The scheduler triggers periodic tasks such as daily or hourly jobs, and socketio manages real-time communication features like notifications. If the worker process crashes, background jobs remain in the Redis queue and will not execute until the worker process is restarted.

Part D – DocStatus Transitions
1. What are the three numeric values of docstatus and what does each represent?

Frappe uses a numeric field called docstatus to represent the lifecycle state of a document.

0 – Draft: The document is still editable and not finalized.

1 – Submitted: The document is finalized and usually represents a confirmed transaction. Editing is restricted.

2 – Cancelled: The document has been cancelled and is considered inactive in the system.

These states help enforce data integrity and control document workflows.


2. Can you call doc.save() on a submitted document? What about doc.submit() on a cancelled one?

Calling doc.save() on a submitted document (docstatus = 1) normally raises an error because submitted documents are locked from modification unless explicitly allowed using doc.flags.ignore_validate_update_after_submit = True.

Calling doc.submit() on a cancelled document (docstatus = 2) is not allowed. Once a document is cancelled, it cannot be submitted again. The correct workflow would usually require creating a new document or amending the cancelled one if the DocType allows amendments.

Frappe enforces these restrictions to maintain transaction integrity and audit history.

3. Why would you see “Document has been modified after you have opened it” and how does Frappe prevent concurrent overwrites?
This error occurs when two users edit the same document simultaneously. If one user saves the document first, the modified timestamp in the database changes. When the second user attempts to save their version, Frappe detects that the document has been modified since it was loaded.
Frappe prevents concurrent overwrites by comparing the modified timestamp of the document in the database with the timestamp stored in the user's session. If they differ, Frappe raises this error to prevent accidental data loss.

3. Unique constraint: DocType "Unique" vs frappe.db.exists() in validate()

There are two common ways to ensure uniqueness in Frappe:

Using the Unique property in DocType

Using frappe.db.exists() in validate()

This method checks manually in Python code whether a record with the same value already exists before saving. While it can prevent duplicates in most cases, it is not completely safe because two requests running at the same time might pass the check and still create duplicates.


D2-Using frappe.get_all() inside a whitelisted method that is accessible to Guest users or low-privilege users can create a security risk. The reason is that frappe.get_all() bypasses standard permission checks by default and directly queries the database. If such a method is exposed through /api/method/..., an unauthorized user may be able to retrieve sensitive records from the system.

Frappe normally protects data access through role-based permissions and permission_query_conditions, which automatically restrict what records a user can see. However, when developers use frappe.get_all() improperly in exposed APIs, those permission filters may not be applied. This means the user could access documents they normally should not be able to view.

To avoid this issue, developers should either use frappe.get_list(), which respects user permissions and permission_query_conditions, or manually enforce permission checks inside the whitelisted method before returning data. This ensures that low-privilege users only receive records they are authorized to access.

E3-Why doc_events is Safer than override_doctype_class

In Frappe, doc_events and override_doctype_class are two different ways to customize the behavior of a DocType, but doc_events is generally safer for most use cases.

doc_events allows developers to attach custom functions to specific document lifecycle events such as validate, before_save, on_submit, or on_cancel. This approach extends the existing behavior without replacing the original DocType logic. The core functionality of the DocType continues to run normally, and the custom logic simply runs before or after those events.

On the other hand, override_doctype_class replaces the entire Python class that implements the DocType. This means the original logic written by Frappe or another app is overridden completely. If the override does not correctly replicate the original behavior, it can break built-in validations, workflows, or other integrations.

Because doc_events adds behavior without replacing the core class, it is safer, easier to maintain, and less likely to break when Frappe or other apps are updated. Therefore, doc_events is recommended for most customizations, while override_doctype_class should only be used when a full replacement of the DocType logic is absolutely necessary.

F5 -_Fieldname Collision Risk

A fieldname collision occurs when a Custom Field created in an app uses the same fieldname as a field that is later added by a Frappe core update or another app.

If this happens, several problems can occur:

Database conflicts – Both fields attempt to create a column with the same name in the same table.

Migration errors – During bench migrate, Frappe may fail because the column already exists.

Unexpected behavior – The system may use the wrong field definition, causing incorrect validation, UI issues, or data inconsistencies.

For example, if a developer adds a Custom Field called priority to a DocType and a future Frappe version also introduces a field named priority, the database schema will conflict.

To avoid this, developers should use unique and app-specific fieldnames, often by prefixing them (e.g., qf_priority) to reduce the chance of collisions with future updates.

1H3-. Making a frappe.call inside the validate client event (before_save handler)

If you try to make a frappe.call (asynchronous server call) inside the client-side validate event or a before_save handler, it does not work as expected.

Why:

The validate event is synchronous and must complete before the document can be saved.

frappe.call is asynchronous (it runs in the background and returns a promise or callback later).

The save process does not wait for the asynchronous call to complete.

As a result:

The document may be saved before the server response is received.

Any validation or changes from the server call may be ignored or applied too late.

Key takeaway: Async calls like frappe.call should not be used inside validate/before_save for blocking validation logic. Use them only for non-blocking updates or notifications.

2. Using onload or refresh for async data fetch

The recommended pattern for fetching data asynchronously on the client side is to use onload or refresh events of the DocType form.

onload runs when the document form is loaded.

refresh runs every time the form is refreshed, e.g., after save or reload.

I4--In Frappe, Prepared Reports and Script Reports serve different use cases based on performance and data freshness requirements. A Prepared Report precomputes its results and stores them in the database, making it fast to display even for large datasets. It is ideal for reports that are queried frequently but do not require real-time accuracy, such as monthly sales summaries or periodic performance dashboards. The main tradeoff is staleness: because the report is generated at a specific point in time, any changes in the underlying data after preparation are not reflected until the report is regenerated. Users may therefore see outdated information if they rely on the prepared output without knowing when it was last computed.

In contrast, a Script Report computes its results on demand, executing Python code or SQL queries in real-time when the user opens the report. This ensures that the data is always current and reflects the latest changes, making it suitable for operational reports where accuracy is critical. However, the tradeoff is performance: for large datasets, generating results in real-time can be slower and may put more load on the server.

I5--Report Builder in Frappe is appropriate when end-users or consultants need a flexible, no-code way to create simple reports based on existing DocType fields. It allows users to select columns, apply filters, and group data without writing any code. Report Builder is ideal for lightweight operational reporting or ad-hoc analysis where the dataset is relatively small, the data structure is simple, and performance is not a critical concern.

A Script Report, on the other hand, is necessary when the report requires complex logic, joins across multiple DocTypes, dynamic calculations, or data transformations that cannot be handled through the Report Builder interface. Script Reports are implemented using Python or SQL and provide complete control over the query and output.

Using Report Builder in a production environment can be a mistake when reporting on large datasets or performance-sensitive operations. For example, generating a Report Builder report on all Job Card entries for a large service company with thousands of records could cause slow page loads or even timeouts because each user request triggers live queries on the database without caching or optimization. In such cases, a Script Report or Prepared Report is the safer and more efficient choice, ensuring proper handling of large data volumes and maintaining application performance.

K1-Frappe uses three background job queues—default, long, and short—to organize and prioritize task execution. The default queue handles standard background jobs that are neither extremely quick nor very time-consuming, such as sending routine email notifications or processing moderate-sized updates. The long queue is intended for heavy or time-consuming tasks, like generating monthly revenue reports, bulk data processing, or large file exports. Jobs in this queue can have a higher timeout to ensure completion without being terminated prematurely. The short queue is for lightweight, fast tasks that need immediate execution, such as updating a single field, small calculations, or minor background actions, so they are not delayed behind larger jobs.

Choosing the correct queue ensures efficient resource allocation and prevents slower tasks from blocking quick operations, keeping the system responsive and reliable. For example, a report generation that may take several minutes should be enqueued on the long queue, while a small update to a document should use the short queue.


K2-In Frappe, you can disable the scheduler for a specific site by setting "disable_scheduler": true in that site’s site_config.json file. This is often done on a development or test site to prevent automatic tasks, such as email notifications, report generation, or background jobs, from running unintentionally and affecting real data. Disabling the scheduler on a dev site allows developers to test code changes safely without triggering production-style automated tasks.

If the scheduler or worker process is down, any scheduled jobs that were queued during that downtime do not get lost. Once the worker restarts, the queued jobs are picked up from Redis and executed in order. This ensures that background tasks are eventually processed even if there is a temporary outage, maintaining consistency and reliability in task execution.

L1-Frappe supports two main authentication methods: session cookie authentication and token-based authentication. Session cookie auth works by logging in a user through the browser, which creates a session on the server and stores a session cookie in the browser. Subsequent requests automatically include this cookie, allowing the server to identify the logged-in user and enforce permissions. This method is ideal for browser-based interactions, such as Desk or Portal usage, because it integrates with CSRF protection and automatically manages the user session.

Token-based auth, on the other hand, uses an API key and secret to authenticate requests without relying on a browser session. Each request includes the token in headers, and the server validates it to identify the user or system making the call. This approach is suitable for server-to-server integrations, background scripts, or external applications, where maintaining a browser session is impractical. Token auth is stateless, more secure for automated processes

Setting allow_guest=True on a Frappe endpoint makes the API accessible to unauthenticated users, which carries real security risks. The main danger is that sensitive data or operations may become exposed to anyone on the internet if proper permission checks are not implemented. Three specific attack vectors include:

M2-Frappe uses Redis as a caching layer to improve performance and reduce database load. Several types of data are cached in Redis to speed up requests and system operations. Five key things that Frappe caches are:

Bootinfo: This includes initial data sent to the client when a user logs in, such as the list of accessible modules, system settings, and user details. Caching bootinfo reduces repeated database queries on each page load.

DocType metadata / meta: Frappe caches DocType definitions, field metadata, and configurations, allowing forms and reports to render quickly without fetching schema information from the database repeatedly.

Website context: Frequently used data for rendering web pages, such as settings, menus, and dynamic content, is cached in Redis to speed up page load times for public and portal pages.

Translations: Language translations for labels, messages, and DocType fields are cached to avoid repeated lookups of .csv translation files during rendering of forms, print formats, or web pages.

User permissions: Role-based permissions and permission query conditions are cached to ensure fast access control checks for each request, preventing unnecessary database queries while enforcing security rules.
